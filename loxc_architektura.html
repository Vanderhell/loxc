Technická dokumentácia · v0.1-draft

loxc

Hierarchický vektorový kompresný formát s dynamickými filtračnými maticami natrénovanými per-doménu.

⚡ Návrh architektúry · Experimentálna fáza
01

Základná myšlienka

Namiesto klasického Huffmanovho stromu (pointer chasing) používa loxc hierarchiu vnorených filtračných matíc rôznych geometrií. Navigácia je čisté indexovanie – žiadne skoky po pamäti, len matrix[x][y][z].

O(1) dekódovanie / znak
~1-4ns L1 cache hit
0B overhead tabuľky
02

Formát súboru

┌──────────────┬────────┬────────┬────────┬─────────────────┬──────────────┐
│ MAGIC  4B    │ VER 1B │ MOD 1B │ LEN 2B │ LOCAL_DICT      │ DÁTA          │
└──────────────┴────────┴────────┴────────┴─────────────────┴──────────────┘
  ↑                                          ↑                 ↑
  identifikuje modul                         zakódovaný        bitový stream
  + jeho matice v knižnici                   vektormi          vektorov

MAGIC

Identifikuje modul a jeho natrénované matice. Tabuľka sa nikdy neposiela – je zabudovaná v knižnici. Magic je kľúč.

0xLXC1  →  anglický text
0xLXC2  →  zdrojový kód
0xLXC3  →  slovenčina
0xLXC4  →  JSON / XML
0xLXCN  →  vlastný modul

LOCAL_DICT

Opakujúce sa slová špecifické pre daný dokument. Zakódované vektormi toho istého modulu. Ide v hlavičke.

ZISK = (N × bits_slova)
- (N × bits_ref + bits_v_dict)

ak ZISK > 0 → do dict

04

Kódovanie – variabilné vektory

Výsledný súbor je bitový stream variabilných vektorov. Časté znaky = krátky vektor = málo bitov. Zriedkavé = hlbšia hierarchia = viac bitov.

Vstup:  "the quick"

t  →  úroveň 1  →  010/011         (6 bitov)
h  →  úroveň 1  →  010/100         (6 bitov)
th →  bigram    →  001             (3 bity – kratší!)
e  →  úroveň 1  →  000/001         (6 bitov)
   →  úroveň 1  →  00              (2 bity – najčastejší!)
ž  →  úroveň 2  →  110/010/011/001 (12 bitov – zriedkavý)
SKORE = frekvencia / (hĺbka ^ penalizácia)

Trénovací parameter penalizácia ovplyvňuje hĺbku stromu matíc – a teda veľkosť v cache.

05

Trénovací pipeline

Čisté vzorky
bez komentárov, dead code, šumu, duplikátov
Frekvenčná analýza
znaky, bigramy, trigramy → distribúcia
Zoskupenie do kategórií
malé/veľké písmená, číslice, symboly, bigramy...
Optimalizácia geometrie matíc
hľadá optimálny tvar (2D/3D, mocniny 2) pre každú skupinu
Výstup → .h súbor
statické C polia, vbudované do modulu, nikdy sa neposielajú
06

Knižnica – C99 modulárna

// Programátor includuje len čo potrebuje
#include "loxc_base.h"        // jadro, vždy povinné
#include "rng_table_en.h"    // anglický text   → magic 0xLXC1
#include "rng_table_code.h"  // zdrojový kód    → magic 0xLXC2
#include "rng_table_sk.h"    // slovenčina      → magic 0xLXC3
#include "rng_table_json.h"  // JSON / XML      → magic 0xLXC4

// API
rng_encode(input, LOXC_PLAIN, output, &len);
rng_decode(input, len, output);
loxc_train(samples, n, .penalizacia = 1.5);

Distribúcia modelov

Natrénovaný model = sada matíc. Ktokoľvek s rovnakým modelom vie kódovať aj dekódovať. Modely sa distribuujú ako malé .h súbory. Komunita môže tvoriť vlastné modely pre špecifické domény (medicína, právo, finančné dáta...).

07

Fáza 2 – Tokenizácia (neskôr)

Voliteľný modul pred hlavným kódovaním. Skráti dlhé tokeny na krátke aliasy:

function  →  f
return    →  r
var input →  v i

Tabuľka aliasov ide spolu s dátami (zakódovaná vektormi).
08

Čo je originálne

Existujúce riešenia loxc
Huffman strom – pointer chasing Filtračné matice – priame indexovanie
Fixná štruktúra stromu Variabilná geometria určená tréningom
Tabuľka sa posiela s dátami Tabuľka v knižnici, magic ju identifikuje
LZ77 sliding window za behu LOCAL_DICT predspracovaný, len vector lookup
Jeden univerzálny slovník Per-doména + per-dokument LOCAL_DICT
Fixné dimenzie 2D aj 3D matice, rôzne veľkosti per vetva
09

Očakávaný výkon

Dekódovanie

L1 cache hit:  ~1-4 ns / znak
L2 cache hit:  ~5-12 ns / znak
L3 cache:      ~20-40 ns / znak

Kľúč: matice musia sediť do L1
Penalizácia hĺbky to garantuje

Zložitosť

Kódovanie:    O(n)
Dekódovanie:  O(n)
Znak lookup:  O(1)
LOCAL_DICT:   O(n)  (hash table)
Tréning:      O(n)  (raz, offline)

⚡ Pre krátke stringy (pod 200 znakov) kde zstd trpí na overhead hlavičky môže loxc výrazne vyhrať – tabuľka je v knižnici, žiadny bootstrap čas.

10

N-gramy – analyzátor rozhodne

Okrem bigramov podporuje loxc trigramy, tetragramy a viac. Analyzátor a generátor rozhodnú sami čo sa oplatí – ty nič nenastavuješ manuálne.

Analyzátor vyskúša všetky n-gramy až po max_n:

"th"    bigram    frekvencia 3.2%  ZISK +8 bit/výskyt   ✓ do tabuľky
"the"   trigram   frekvencia 2.1%  ZISK +12 bit/výskyt  ✓ do tabuľky
"the "  tetragram frekvencia 1.8%  ZISK +14 bit/výskyt  ✓ do tabuľky
"tion"  tetragram frekvencia 1.4%  ZISK +11 bit/výskyt  ✓ do tabuľky
"ation" pentagram frekvencia 0.9%  ZISK +6 bit/výskyt   ✓ do tabuľky
"xyz"   trigram   frekvencia 0.01% ZISK -2 bit/výskyt   ✗ zahodiť

Hranica: tabuľka vs LOCAL_DICT

TypKdePrečo
Krátke n-gramy (2–6 znakov) tabuľka v knižnici globálne časté, natrénované raz
Dlhé opakujúce sa slová (7+ znakov) LOCAL_DICT v hlavičke lokálne časté, špecifické pre dokument 
// Tréning s parametrami
loxc_train(samples, n,
    .max_ngram   = 6,      // skúsi až 6-znakové sekvencie
    .penalizacia = 1.5     // penalizácia hĺbky matíc
);
11

Predpripravené moduly – stavebné kamene

loxc nerozlišuje jazyky – rozlišuje vlastnosti textu. Skladáš len čo tvoje dáta skutočne obsahujú. Každý modul je samostatná kompilačná jednotka – linker vyhodí čo neincluduješ.

// ── Základné kamene ──────────────────────────────────────
#include "loxc_base.h"         // jadro – vždy povinné

// Znaky
#include "loxc_lower.h"        // malé písmená bez diakritiky   a–z
#include "loxc_upper.h"        // veľké písmená bez diakritiky  A–Z
#include "loxc_lower_dia.h"    // malé písmená s diakritikou    á é í...
#include "loxc_upper_dia.h"    // veľké písmená s diakritikou   Á É Í...
#include "loxc_digits.h"       // číslice                       0–9
#include "loxc_punct.h"        // základná interpunkcia         .,!?;:'"-
#include "loxc_symbols.h"      // programátorské symboly        @#$%^&*()[]{}\/
#include "loxc_whitespace.h"   // medzera, tab, newline, CR

// N-gramy (natrénované vzory)
#include "loxc_ngram_plain.h"  // časté sekvencie bez diakritiky (the, in, er...)
#include "loxc_ngram_dia.h"    // časté sekvencie s diakritikou  (ou, an, ie...)
#include "loxc_ngram_code.h"   // časté sekvencie v kóde        (if, == , ->...)
#include "loxc_ngram_data.h"   // časté sekvencie v dátach      ({", ": , true...)

// ── Hotové kompozície ────────────────────────────────────
#include "loxc_plain.h"        // lower+upper+digits+punct+ngram_plain
#include "loxc_plain_dia.h"    // + lower_dia+upper_dia+ngram_dia
#include "loxc_code.h"         // lower+upper+digits+symbols+ngram_code
#include "loxc_data.h"         // lower+digits+punct+symbols+ngram_data

// ── Vlastný modul (ty / komunita) ────────────────────────
#include "loxc_mymodule.h"     // natrénovaný na tvojich dátach

Bez diakritiky vs s diakritikou

Angličtina, kód, JSON, CSV – nikdy nepotrebujú diakritiku. Prečo za ňu platiť miestom v matici? rng_plain.h ju neobsahuje. rng_plain_dia.h ju pridá len ak skutočne treba.

Univerzálnosť cez kompozíciu

Namiesto "EN modul" alebo "SK modul" skladáš vlastnosti. Text v slovenčine = plain + dia. Text v kóde = code. Mix = oba. Jazyk nie je parameter – je to dôsledok výberu modulov.

11b

Príklady nasadenia

Ako loxc navrhnúť pre konkrétne reálne scenáre.

IoT · MQTT

Senzorové správy na obmedzenom zariadení

MQTT správy z IoT senzorov majú fixnú štruktúru – topic, payload s číslami a kľúčmi. Opakujú sa tisíckrát denne. Ideálny kandidát pre loxc s vlastným natrénovaným modulom.

Typická MQTT správa (opakuje sa)
topic: sensors/room1/temp
{
  "device": "TH-204",
  "temp": 21.4,
  "humidity": 58,
  "ts": 1746878400
}
Navrhované riešenie
// 1. Natrénuj modul na vzorkách
loxc_train(mqtt_samples, n,
  .max_ngram = 4);
// → loxc_mqtt.h  (pár KB v ROM)

// 2. Na zariadení len decode
#include "loxc_base.h"
#include "loxc_digits.h"
#include "loxc_ngram_data.h"

// 3. "device","temp","humidity"
//    → LOCAL_DICT → 1B referencie
// Úspora: ~60-70% · RAM: <4KB

IoT zariadenie obsahuje len rng_base + rng_digits + natrénovaný modul. Žiadna generická knižnica, žiadny bloat. Dekóder zaberie <4KB ROM – funguje aj na ATmega328.

Backend · REST API

Kompresia JSON odpovedí bez overhead

REST API vracia podobne štruktúrované JSON odpovede. gzip/brotli majú overhead pre krátke payloady. loxc s natrénovaným modulom na tvojom API schéme ich porazí.

Situácia
// Každý endpoint vracia
// podobnú štruktúru

GET /users      → user objekt
GET /products   → product objekt
GET /orders     → order objekt

// Kľúče sa opakujú v každej
// odpovedi: "id","name",
// "created_at","status"...
Navrhované riešenie
// Natrénuj na svojom API schéme
loxc_train(api_responses, n);
// → loxc_myapi.h

// Server aj klient includujú
// rovnaký modul
#include "loxc_base.h"
#include "loxc_myapi.h"

// "created_at" → 2B kód
// "status":"active" → 3B
// Úspora: ~65% · latency -40%
Embedded · Logging

Logy na zariadení s obmedzenou flash pamäťou

Embedded zariadenie loguje chyby a udalosti do flash pamäte. Každý ušetrený bajt = dlhší log históriu pred prepísaním.

Typické log správy
ERR  sensor_read timeout 3
WARN battery_low 18%
INFO boot_complete v2.1.4
ERR  network_conn failed
INFO ota_update start
Navrhované riešenie
// Modul natrénovaný na
// znaych log správach
#include "loxc_base.h"
#include "loxc_lower.h"
#include "loxc_digits.h"
#include "loxc_mydevice_log.h"

// "ERR","WARN","INFO" → 2 bity
// "_" → 1 bit (veľmi časté)
// Flash využitie -55%
// = 2× dlhšia história logov
Real-time · Chat protokol

Vlastný binárny protokol pre správy

Vlastný chat protokol kde každý bajt znižuje latency. loxc ako transportná vrstva – správy sa kódujú pred odoslaním, dekódujú po prijatí.

Packet štruktúra
// Bez loxc (raw JSON)
{
  "to": "user42",
  "msg": "Hey, are you there?",
  "ts": 1746878400
}
= 68 bajtov / správa

// S loxc
= ~28 bajtov / správa
= -59% bandwidth
Navrhované riešenie
// Server aj klient majú
// rovnaký modul (magic)
#include "loxc_base.h"
#include "loxc_plain.h"
#include "loxc_ngram_data.h"

// Kódovanie pred odoslaním
loxc_encode(msg_json,
           LOXC_PLAIN, out, &len);

// Dekódovanie po prijatí
loxc_decode(packet, pkt_len,
           msg_json);

⚡ Keďže modul je v knižnici (magic), kódovanie začína okamžite bez handshaку – ideálne pre real-time protokoly kde každá milisekunda rozhoduje.

Databáza · Storage

Kompresia stĺpcov s opakujúcimi sa hodnotami

Databázové stĺpce ako "status", "country", "category" majú malý počet opakujúcich sa hodnôt. loxc s natrénovaným modulom na schéme tabuľky ich komprimuje extrémne efektívne.

Tabuľka orders (milióny riadkov)
status:   "pending" | "shipped"
          | "delivered" | "cancelled"
country:  "SK" | "CZ" | "AT" | "DE"
category: "electronics" | "clothing"
          | "food" | "books"
Navrhované riešenie
// Natrénuj na hodnotách schémy
loxc_train(schema_values, n);

// "pending"   → 3 bity
// "delivered" → 4 bity
// "SK"        → 2 bity
// Milión riadkov: -70% storage
// Dekódovanie: ~2ns / hodnota
12

Dizajnová filozofia

Modularita

Každý modul rieši jeden problém. Žiadny bloat. Includneš len čo skutočne potrebuješ – zvyšok neexistuje pre tvoj projekt.

// Chceš len angličtinu?
// Dostaneš len angličtinu.
#include "loxc_base.h"
#include "loxc_plain.h"

Linker friendly

Každý modul je samostatná kompilačná jednotka. Linker vyhodí čo nepoužívaš. Kritické pre embedded a IoT kde každý bajt rozhoduje.

rng_base.o    ~2.0 KB
rng_lower.o   ~0.5 KB
rng_digits.o  ~0.2 KB
rng_upper.o   ← vyhodené linkerom
─────────────────────
Spolu:        ~2.7 KB
zstd:          ~400 KB

Len knižnica – nič viac

loxc rieši jeden problém: kompresia a dekomprimácia. Čo s tým urobíš je tvoja vec – CLI nástroj, sieťový protokol, databázový engine, IoT firmware. Rovnako ako zlib, sqlite, alebo openssl – len knižnica, nad ktorou každý stavia čo potrebuje.

13

Dev nástroje (nie súčasť knižnice)

Dva standalone nástroje pre vývojárov ktorí s knižnicou pracujú:

loxc_train

Trénovací nástroj. Vezme vzorky, analyzuje frekvencie, optimalizuje geometrie matíc, vygeneruje .h modul.

loxc_train \
  --input  samples/ \
  --magic  0xLXC5 \
  --ngram  6 \
  --output rng_mymodule.h

loxc_cli

Jednoduchý CLI nástroj na testovanie a ladenie. Nie pre produkciu – len pre vývojárov.

loxc encode --magic en file.txt \
  > file.lxc

loxc decode file.lxc

int loxc_encode(
    const char *input,
    uint32_t    magic,
    uint8_t    *output,
    size_t     *out_len)
{
    if (!input || !output)
        return LOXC_ERR_NULL;
    return loxc_encode_impl(
        input, magic,
        output, out_len);
}

Otvorené otázky a návrhy riešení

Veci ktoré treba doriešiť pred produkčným nasadením.

Streaming / chunking

Neriešené

Čo ak je vstup príliš veľký na to aby bol celý v pamäti? LOCAL_DICT potrebuje vidieť celý vstup pred kódovaním – to je problém pre 10GB log súbor.

// Navrhované riešenie: dvojfázový streaming

// Fáza 1: priechod len pre LOCAL_DICT analýzu
loxc_stream_t *s = loxc_stream_open(LOXC_PLAIN);
loxc_stream_feed(s, chunk, len);   // len počíta frekvencie
loxc_stream_feed(s, chunk2, len2); // ...
loxc_stream_finalize_dict(s);      // zostav LOCAL_DICT

// Fáza 2: reálne kódovanie s hotovým dict
loxc_stream_encode(s, chunk, len, out, &out_len);
loxc_stream_close(s);
Trade-off: 2× priechod vstupom. Alternatíva: obmedzený LOCAL_DICT len pre prvých N znakov (sliding window).

Paralelné kódovanie

Riešiteľné jednoducho

Matice sú read-only po natrénovaní – žiadny shared state. Viacero vlákien môže kódovať súčasne bez zamykania. Treba to len explicitne navrhnúť a dokumentovať.

// Matice sú konštanty – thread safe automaticky
// Každé vlákno má vlastný output buffer
// Žiadne mutex, žiadne atomic – len čisté čítanie

// Thread 1                Thread 2
loxc_encode(in1, ...);     loxc_encode(in2, ...);
// Obe pristupujú k tým istým maticiam – OK
Pozor: LOCAL_DICT analýza (hash table) nie je thread safe – každé vlákno potrebuje vlastnú inštanciu.

Verzia modulu – spätná kompatibilita

Neriešené

Magic identifikuje modul ale nie verziu matíc. Čo ak natrénuješ lepší model neskôr? Staré dáta zakódované v0 musia byť čitateľné knižnicou s v1 modulom.

// Navrhované riešenie: magic obsahuje verziu

// Štruktúra magic (4 bajty):
[ LX ][ C ][ MODULE_ID ][ VERSION ]
  2B    1B      1B           1B

// Príklady:
0x4C58 0x43 0x01 0x01  → loxc, plain modul, verzia 1
0x4C58 0x43 0x01 0x02  → loxc, plain modul, verzia 2

// Knižnica udržiava všetky verzie modulov
// Decoder vyberie správnu podľa magic
255 verzií na modul – v praxi viac než dosť. Staré verzie sa nikdy neodstránia, len označia ako deprecated.

Bezpečnosť dekodéra – malformovaný vstup

Kritické

Čo ak prídu poškodené alebo zámerné malformované dáta? Decoder musí byť odolný – žiadne buffer overflow, žiadne nekonečné slučky.

// Navrhované opatrenia:

// 1. Vždy kontroluj LEN z hlavičky
if (in_len < LOXC_HEADER_SIZE) return LOXC_ERR_TRUNCATED;

// 2. Max hĺbka matice je known at compile time
//    → žiadna nekonečná slučka možná

// 3. Output buffer vždy s explicitnou veľkosťou
loxc_decode(in, in_len, out, out_max, &out_len);

// 4. Fuzzing ako súčasť CI pipeline
//    AFL++ / libFuzzer na decoder
Keďže matice sú fixné a hĺbka je ohraničená tréningom, decoder má prirodzene obmedzené správanie – lepšia situácia než LZ77 sliding window.

Integrita dát – checksuma

Riešiteľné jednoducho

Jednoduchý CRC32 v hlavičke – decoder overí integritu pred dekódovaním. Voliteľné, ale odporúčané pre sieťový prenos a ukladanie.

// Rozšírený formát hlavičky s CRC:
[ MAGIC 4B ][ VER 1B ][ MOD 1B ][ LEN 2B ][ CRC32 4B ][ DICT ][ DATA ]

// CRC32 pokrýva: DICT + DATA (nie hlavičku)
// Voliteľný flag v MOD byte:
#define LOXC_FLAG_CRC  0x80  // bit 7 = CRC prítomný
#define LOXC_FLAG_DICT 0x40  // bit 6 = LOCAL_DICT prítomný

// Decoder:
if (header.flags & LOXC_FLAG_CRC)
    loxc_verify_crc(data, len, header.crc);
CRC32 overhead: 4 bajty. Pre IoT kde každý bajt rozhoduje – voliteľné vypnutie flagom.

Validácia výkonu – benchmark suite

Súčasť dev nástrojov

Kým nespustíme reálne benchmarky, všetky výkonnostné čísla sú odhady. Treba benchmark suite ktorý porovná loxc vs zstd vs lz4 vs gzip na reálnych dátach.

loxc_bench   --input  samples/   --rivals zstd,lz4,gzip   --report bench_report.html

// Výstup: kompresný pomer, ns/znak,
// cache hit rate, pamäťová stopa
// pre každý formát a každého rivala
Benchmark je tretí dev nástroj vedľa loxc_train a loxc_cli. Výsledky benchmarku informujú ďalší tréning.
